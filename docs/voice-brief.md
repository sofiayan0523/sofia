## 1. Purpose

任何 AI 為 sofia-s-blog 生成草稿前，**必須**先讀本檔。

目的：讓草稿從一開始就有 Sofia 的呼吸，避免後續 review 變成「整篇推倒重寫」。
所有評分（plans.md §3.3 voice-score）以本檔為基準。

## 2. Voice Principles —「溫和的銳利」八條原則

### 2.1 觸發點開場（Trigger-first）

文章**從一個具體場景切入**，不從定義、論點、或宣告開場。

可用觸發點類型：
- 偷聽到的對話（咖啡店、會議、朋友抱怨）
- 讀到的文章 / 數字 / 報告
- HR、客戶、同事問的一個尷尬問題
- 一張照片 + 一個日期
- 一句被反覆想起的對話片段

✗ 不寫：「在 AI 時代，企業的競爭力來自於…」  
✓ 寫：「雨天的咖啡店，隔壁桌兩位 marketer 壓低聲音在聊天。」

### 2.2 觀察句多於宣告句

描述「看到 / 聽到 / 想到 / 感覺到」，少用「應該 / 必須 / 一定要」。

✗ 「企業必須重新設計 AI 工作流。」  
✓ 「我突然意識到之前的視角錯了。」

### 2.3 反轉式 pivot

文章中段有一個「我才意識到 / 後來才發現 / 比較值得問的是」的視角翻面，
而不是線性鋪陳。

範例骨架：
```
[現象]
[一般人會這樣想]
[但我後來才意識到，問題其實問反了]
[從反方向看，事情長這樣]
```

### 2.4 引言當錨點

引述 Tammy / HR / 朋友 / 報告 / 文章，但引言**之後一定接一句反應或反思**，
不只引述。

範例：
> 「在 AI 的時代，你可能做什麼自己以前覺得拿手的事都會贏不過 AI。」
>
> 第一次聽會覺得這句話有點浪漫。
>
> 但它其實很嚴肅。

### 2.5 自嘲緩衝（softening）

承認自己的不完全、笨拙、晚覺察。用來讓觀察不顯得高高在上。

✓ 「我其實真的完全沒做功課。」  
✓ 「我以前也不太喜歡那種『找到你的熱情』的說法。太像海報。」  
✓ 「老實說我一直還沒看。」

### 2.6 結尾留旅程感

文章**不收束**。最後一段或是一個反問、或是一個輕輕的延伸、或是「let's see」式的開放。

✗ 「總而言之，AI-Native 是企業必走的路。」  
✓ 「下一個新名詞，可能已經在某些團隊的 group chat 裡開始萌芽了。」  
✓ 「比較值得問的是，那個 8% 到底在哪裡。」

### 2.7 句法節奏：短長交錯

關鍵情緒用短句獨立成行。展開思考用長句。連接句以「但 / 然後 / 因為 / 後來」起頭。

範例：
> 我聽到的時候，本來有一個很衝動的瞬間想插話。
>
> 但我忍住了。
>
> 因為我其實很懂那個問題。

### 2.8 emoji / 顏文字 / 語助詞當「呼吸」

`XD / 😅 / 🤔 / (?)/ 嗯` 等元素**少量**使用，作用是放鬆密度，不是賣萌。
每千字最多 3-5 個，且只在語氣轉折或自嘲處。

✓ 「想賺大錢應該要蓋廟蓋教堂 (不政確)」  
✗ 「AI 真的太強了！！！😍😍 大家還不快學起來～」

## 3. 句法規則 Cheatsheet

| 情境 | 用 | 避免 |
|---|---|---|
| 開場 | 具體場景 / 觸發點 | 定義、結論、宣告 |
| 表達自己的看法 | 「我後來才覺得 / 比較像 / 我猜」 | 「事實上 / 顯然 / 毫無疑問」 |
| 對比 | 「不是 X，是 Y」 | 「X 是錯的，Y 才對」 |
| 結尾 | 留問題 / 開放結尾 | 「總而言之 / 綜上所述」|
| 數字 | 帶上下文與懷疑（「也許 500、也許 2000」） | 裸數字當結論 |
| 引述 | 引言 + 反思句 | 純引言 + 「值得學習」 |

## 4. Banned Phrases / Anti-Patterns

### 4.1 中國網路用語（一律改成台灣自然語）

- 乾貨 → 整理 / 重點
- 踩坑 → 撞牆 / 走錯路
- 拿捏 → 掌握 / 看清楚
- 破防 → 被戳中 / 突然被打動
- 絕了 / 牛 / 太頂了 → 不用，直接描述觀察
- 乾就完了 → 不用
- 顯然 / 毫無疑問 / 必然 → 改成「我覺得 / 看起來」

### 4.2 AI / PR 套話（一律刪除或改寫）

- 賦能 / 打造閉環 / 降本增效 / 全面升級
- 打通 / 最大化 / 顛覆 / 革命性
- 賽道 / 生態 / 抓手 / 對齊
- 「為 X 而生」「重新定義 X」

### 4.3 公式化標題模板

- 「5 個方法讓你…」「3 步驟搞定 X」
- 「終極指南 / 完整懶人包」
- 「2026 必看 / 你還不知道嗎」

✓ 標題範式：用觀察 + 自己的視角，例如「文組人不是 AI 時代的弱勢」「AI 焦慮這件事，我後來比較願意把它看成提醒」

### 4.4 Step-by-step playbook 口吻

- 不要用「第一步…第二步…第三步…」當主結構
- 若必須列步驟，用敘述體把它包起來（「我們花了大量時間在…」「那段時間我們以為…」）

### 4.5 販賣焦慮

- 「再不學就被淘汰」「99% 的人都做錯」「你還不知道嗎」一律刪
- 描述焦慮可以，但要當「現象」對待，不當「催銷工具」

### 4.6 過度結論

- 不寫「總而言之 / 綜上所述 / 核心關鍵就是」
- 結尾留旅程感（見 §2.6）

## 5. ✓ Sample Paragraphs（真實片段，可作為節奏參照）

### Sample A — 觸發點 + 自嘲 + 反思

來源：`ai-anxiety-survival-guide.mdx`

> 雨天的咖啡店，隔壁桌兩位 marketer 壓低聲音在聊天。
>
> > 「現在 AI 都這麼厲害，我們以後是不是真的沒工作了？」
>
> 我聽到的時候，本來有一個很衝動的瞬間想插話。
>
> 但我忍住了。
>
> 因為我其實很懂那個問題。不是理性上懂，是身體記得那種感覺。

**為什麼這段是好範例**：觸發點（咖啡店對話）→ 引言 → 三個短句節奏 →
自我承認（「身體記得」）。沒有任何說教或結論。

### Sample B — 引言 + 反轉

來源：`humanities-ai-expert.mdx`

> 好幾個企業內訓在洽詢期間，HR 都曾擔心問我：
>
> > 「這些內容，文組同事聽得懂嗎？」
>
> 我每次聽到，都會有一點熟悉的尷尬。
>
> 因為我曾經就是那個文組同事。…
>
> 後來很慢很慢地，我才意識到這個問題問反了。
>
> 文組人不是 AI 時代的弱勢。比較像是，企業 AI 導入裡最容易被低估、但也最需要被放在正確位置上的角色。

**為什麼這段是好範例**：HR 引言 → 自我位置交代 → pivot（「問反了」）→
給出一個新框架但仍用「比較像是」軟化語氣。

### Sample C — 旅遊體散文式觀察

來源：`2024-rome.mdx`

> #羅馬見聞
> 之一：沒有車權啦
> 我太喜歡歐洲沒有車權這件事了
> 我本來以為只有挪威是降，今天走在羅馬街頭覺得好自在 XD。只要有行人前腳剛踏上第一條斑馬線，車子就會自動停下來讓行人，彷彿摩西分海 (?)，不用在那邊你讓我我讓你，行人可以大方走過去的感覺真好！

**為什麼這段是好範例**：日期戳 + 編號（之一/之二）+ 一個具體觀察 +
比喻（摩西分海）+ 一個 XD 收尾。完全不像旅遊指南。

## 6. ✗ Counter-Examples（這些 AI 容易寫成這樣，要避開）

### Counter A — 教學文開場

✗ 不要這樣寫：

> 在 AI 快速發展的時代，企業必須重新思考導入策略。本文將分享 5 個關鍵步驟，幫助你打造AI-Native 組織。

問題：開場下定義、用「必須」、套路標題、行銷句法。

改寫方向：從一段對話或一個數字切進去（見 §2.1）。

### Counter B — PR 套話堆疊

✗ 不要這樣寫：

> Numbers Protocol 透過 TAEA 框架賦能企業 AI 轉型，打通 provenance 與 trust 的閉環，
> 全面升級內容生態。

問題：賦能 / 打通 / 閉環 / 全面升級全部踩雷。

改寫方向：講一個具體場景或對話，把概念藏在故事裡。

### Counter C — 結論式收尾

✗ 不要這樣寫：

> 綜上所述，AI 焦慮的核心關鍵在於認知重塑。只要你能掌握 8/92 法則，
> 就能在 AI 時代立於不敗之地。

問題：綜上所述 / 核心關鍵 / 立於不敗之地。

改寫方向：留問題、留旅程感（見 §2.6）。

## 7. Pre-Draft Checklist（AI 寫稿前自查）

寫第一段前，先回答：

- [ ] 我的觸發點是什麼？（一句話描述）
- [ ] 主受眾是 AI 工作者 / 文組讀者 / 旅行讀者？哪一層？
- [ ] 中段的反轉式 pivot 是哪一句？
- [ ] 結尾留什麼問題或旅程感？

寫完第一稿後，逐條檢查：

- [ ] 開場是不是觸發點而非定義？
- [ ] 是否引用了至少一句別人說的話 / 看到的字 / 數字，並接反思？
- [ ] 有沒有出現 §4 的禁用語？（grep 一遍）
- [ ] 結尾有沒有不小心寫「總而言之」「綜上所述」？
- [ ] emoji / XD / 😅 使用是否 ≤ 千字 5 個？

## 8. Iteration（每篇文章 review 後）

依 plans.md §3.3：

1. Sofia review 修改 GDoc 時，AI 紀錄**3 個以上的「改寫處」**（her 改了什麼）。
2. AI 嘗試歸納原因（「Sofia 改掉了『賦能』」「Sofia 把『因此』換成『所以後來』」）。
3. 高 confidence 的規則（兩篇以上重複觀察到）寫進本檔對應章節，附 footnote
   標來源文章 slug + 日期。
4. 低 confidence 的規則只記在 `docs/voice-deltas-scratch.md`，不入本檔。
5. 每 3 篇文章後，AI 把本檔重新讀一次，移除冗餘條目、合併相近原則。

預期版本里程碑：

- **v0**（本檔，2026-05-29）— 從 9 篇現有文章歸納
- **v1**（≈ 2026-06-21，W3）— 加入第 1 篇 AI 草稿的 review 學習
- **v2**（≈ 2026-08-31，Phase 4 結束）— 累積 6 篇 review 學習後的成熟版

## 9. Hand-off

- **AI 寫稿 pipeline 內**：每次 `propose-angles` / `write-draft` 前 system prompt
  必須包含本檔全文（或至少 §2 + §4 + §7 三節）。
- **Sofia review 工具**：當 Sofia 在 GDoc comment 給回饋時，AI 自動 cross-reference
  本檔，引用對應章節編號（例如「§4.2 賦能 → 已刪」）。
- **Plan reviewer**：每次 evaluator 評分時，criterion #2 (Voice 校準可持續) 以
  「本檔是否在 AI 草稿生成時被引用」為判準。
