前幾天看到 Manny 轉了 [Greg Isenberg 一篇關於 AI-Native 公司的長文](https://www.facebook.com/share/p/14bxbEbxTru/)，順手把原文也讀了一遍。讀完心裡只有一個念頭：

> 「這幾乎就是我 6/3 在 AppWorks 想講的東西，但他講得更狠。」

Greg 開頭就丟了一句很扎心的話：

> **「現在每家公司都說自己是 AI-Native。但大部分時候，這只是代表團隊裡有人開著 ChatGPT 的分頁。」**

然後他丟出一個數字：**全球年營收超過 500 萬美金、且真正做到 AI-Native 的公司，大概只有 1,000 家。** 也許 500、也許 2,000，數字不重要，重點是：**這個領域基本上是空的**。

我盯著這個數字看了很久。因為過去這一年我每週都在企業內訓現場、新創 demo day、創投 office hour 看到「我們是 AI-Native」這四個字。但只要稍微深問一句「你們資料怎麼結構化？agent 邊界怎麼定？」，多數人會卡住。

這篇是寫給 6/3 那場 AppWorks 演講的暖身。也是寫給每一位「想做 AI-Native 但不確定具體該做什麼」的 founder。

## AI-Assisted vs AI-Native：一個問題的差別

Greg 文章最值錢的一段是這句對比：

- **AI-Assisted 的公司問**：「我們可以在哪裡加 AI 來省時間？」
- **AI-Native 的公司問**：「如果 agent 負責前 80% 的工作，這個流程應該長什麼樣子？」

**第二個問題改變一切**。

第一個問題的答案是「在現有流程上加個 ChatGPT 分頁」。第二個問題的答案是「**整個流程要重寫**」。

舉客服當例子。一般公司客服工單進來：人讀、查帳戶、回想政策、寫回覆、可能問工程、可能升級、可能忘了標分類。**整個是以人為主導、軟體散在周圍的流程**。

AI-Native 的公司客服工單進來：agent 讀客戶歷史、檢查方案限制、查過去工單、參考政策、擬回覆、建議處理方式，能解決就解決，不能就附上「需要人判斷的原因」送過來。**人的角色不是搜尋引擎、不是路由器、不是文案寫手。人的角色是審核模糊地帶**。

那是一家**完全不同的公司**。

這也是為什麼我寫 [AI 同事方法論](/ai-coworker-methodology) 一直在強調：別買 seat，要指派。買 seat 是 AI-Assisted 思維（讓人類員工多一個工具）。指派是 AI-Native 思維（**讓 agent 變成同事，而你的人類員工去做更值錢的事**）。

## 為什麼大多數公司「對機器來說根本不可讀」

Greg 用了一個很精準的詞：**legibility**（可讀性）。

> **「AI-Native 公司不是一家『有在用 AI』的公司。它是一家被重新建構過的公司，讓 AI 能夠真正在裡面運作。企業的架構、文件、權限、監測方式，全部以 agent 能理解的方式設計。公司讓自己對機器變得可讀。」**

聽起來很無聊。直到你意識到 — 這可能是未來十年最大的商業優勢。

因為大多數公司對機器根本不可讀。**對自己的員工也幾乎不可讀**。

- CRM 說一套
- Slack 討論串說另一套
- 真實的客戶歷史躺在某個人的信箱裡
- 定價邏輯在一份叫 `Final_v7_NEW` 的試算表裡
- 退費政策在一份沒人信任的 Notion 文件裡
- 銷售流程是「去問 Sarah，她知道企業客戶怎麼接」
- 上線流程要用五個工具、經過三個人、兩道審核、外加一個創辦人還是會被拉進來處理邊緣情況

然後這些公司問：「為什麼 AI 不能幫我們做更多？」

**因為 AI 沒辦法靠 vibe 運作**。它無法經營一個真相散落在人員、工具、習慣、例外、組織記憶之間的企業。**Agent 需要上下文。需要乾淨的輸入。需要規則。需要存取權。需要邊界。需要知道什麼叫做「好」。需要知道什麼時候該動手、什麼時候該問人。**

大多數公司花了二十年買軟體，但沒有花二十年設計一個作業系統。**他們有一堆工具，不是一台機器**。

## 廚房就是公司：沒人想做的那些事

Greg 文章最有畫面感的一句：

> **「Everyone wants the magic. Nobody wants to clean the kitchen. But the kitchen IS the company.」**

廚房就是公司。

而 AI-Native 的門檻**不在模型**（每個人都能買到 Claude、GPT、Gemini API），**在這些無聊事**：

1. **清資料**：把散在 30 個系統的客戶紀錄整理成一份 source of truth
2. **寫 SOP**：把「Sarah 知道企業客戶怎麼接」變成可被 agent 讀懂的文件
3. **把決策邏輯從人的腦袋裡搬出來**：退費政策、定價規則、升級門檻、語氣指引
4. **建邊界**：哪些操作 agent 可以自己做、哪些要核准、哪些絕對不能自動化
5. **建稽核軌跡**：每一個 agent 動作要可被審查、可被回溯

這些事不性感。沒有 keynote 可以講。沒有 demo 可以炫。但這些事**就是公司**。

**Not sexy from the outside. Extremely sexy in the bank account.**（這句我打算直接拿去 6/3 當投影片）

## 給 AppWorks Founder 的 5 步 Playbook

Greg 文章後半給了一個非常實用的 5 步框架。我用我們在 Numbers Protocol 實際做過的版本翻譯給你聽：

### Step 1：挑一個窄範圍、有明確經濟價值的流程

不要從「讓公司變 AI-Native」開始。**太抽象**。

選一個量大、有規則、人花太多時間在協調上的流程。例子：客服解決、外發開發、onboarding、合約初稿、報表生成、續約管理。

我們自己當年的第一個切入點是「客戶問題分類 + 標準回覆草稿」。**不是因為它最性感，是因為它最枯燥**。

### Step 2：把流程當成一台機器來拆

問六個問題：

1. **什麼觸發它？**（email / Slack / form submission？）
2. **需要哪些資料？**（客戶歷史、產品狀態、政策文件？）
3. **會出現哪些決策？**（哪些可逆、哪些要核准？）
4. **成功長什麼樣？**（KPI、品質指標、時間目標）
5. **錯誤在哪裡發生？**（過去 3 個月的 incident log）
6. **人知道什麼是系統不知道的？**（這條最關鍵 — 通常是隱性知識）

把這六個問題的答案寫成一份文件。**這份文件就是你的 agent spec**。

### Step 3：結構化知識

Agent 需要政策 → 寫政策。
Agent 需要定價規則 → 把規則明確化。
Agent 需要客戶歷史 → 清理客戶資料。
Agent 需要範例 → 建範例庫。
Agent 需要語氣 → 定義 tone guide。

**大多數團隊在這一步放棄，因為感覺像在寫文件。但這不是文件。這是基礎建設。**

我在 [TAEA framework](/ai-coworker-methodology) 裡反覆強調的「Auditable」就是這層 — 沒有結構化的知識，agent 永遠在猜，你永遠在解釋為什麼 AI 又把客戶得罪了。

### Step 4：把 agent 放進流程，設好邊界

讓 agent 起草、分類、推薦、充實、摘要、準備。**只在風險可控的地方給行動權。需要判斷的地方要求核准。記錄一切。**

具體技術上的對照（用 Omni 平台為例）：

- **可讀的工作記憶** → 每個對話有 `.omni/memory.md`，跨對話有 Shared Memory 五分類
- **可被觸發的 SOP** → Agent Skills（AI 自動偵測何時該套用）
- **多 agent 編排** → Subagents + Heartbeat（4 小時／24 小時定期掃描）+ Monitor（事件驅動背景監控）
- **邊界與審核** → Plan Mode（agent 提案 → 你核准）、confirmation hooks（高風險動作前彈窗）
- **稽核軌跡** → 所有 tool call 即時 streaming，每一筆寫入操作落到 audit log

這不是賣 Omni，這是賣**架構**。你用任何工具實作這些功能都可以 — 但你得知道**這些功能就是 AI-Native 的最小可行架構**。

### Step 5：衡量業務影響

不是「省了幾小時」這種假試算表數字。Greg 的清單我直接照貼：

- 解決時間（resolution time）
- 轉換率（conversion rate）
- 毛利率（gross margin）
- **每人營收（revenue per employee）** ← 這條最關鍵
- 錯誤率（error rate）
- 客戶滿意度（CSAT）
- 銷售速度（sales velocity）
- Onboarding 時間
- 續約率（retention rate）

**AI-Native 的公司應該在這些數字裡看得出來。看不出來就還沒到。**

## 我們自己的數字（以及我為什麼相信這件事）

如果你看過我之前那篇 [《為什麼 95% 企業 AI 導入會失敗》](/posts/why-95-percent-ai-adoption-fails)，你會看到我們 Numbers Protocol 從 2025 年 4 月全面 AI 導入後的狀態：

- **全職人類同事 4 位** + **線上 AI 同事 15 位** + **實習階段 8 位**
- 每位人類同事**平均同時跟 3 位 AI 同事協作**
- 內部 MVP 從需求到完成 **≤ 24 小時**
- 年度 SaaS / Infra 支出 **-60%**

對照 Greg 的判準：

| Greg 的 AI-Native 訊號 | 我們的狀態 |
|---|---|
| 「12 個人做以前 80 個人的事」 | 4 人 + 15 AI 同事，做以前需要 ~30 人的事 |
| 「每位員工帶來的營收成為定義性指標」 | revenue per employee 是我們董事會月報第一條 |
| 「核心工作流程為了讓 agent 執行、人類監督而設計」 | 客服分類、PR review、Heartbeat 健康監測、報表生成全 agent-first |
| 「agent 知道何時動手、何時問人」 | TAEA framework + Z App agent ticket escalation |
| 「每一個重複出現的決策變成決策系統」 | NREM（AI 自己的睡眠：把對話模式整合回 skill / memory） |

我不是在說我們完美。我是在說：**這套架構是真實可行的，不是 thought leadership PPT**。

而且我們的人類團隊**比過去任何時候都更被重要**。一位優秀的營運人員變成 10 個工作流程的主管。一位優秀的客服變成升級邏輯與顧客體驗的設計師。**這是 Greg 文章那句「人的角色會被放大，不是變得不重要」的真實版本**。

## 給 6/3 AppWorks Founder 的 3 個立刻可做

如果你是早期新創的 founder，這週就可以開始的三件事：

### 1. 列出你公司「真相在誰腦袋裡」的清單

打開 Notion 或 Google Doc，認真列：

- 定價邏輯在誰腦袋裡？
- 退費政策在誰腦袋裡？
- 「合格的 lead」定義在誰腦袋裡？
- 客戶歷史散在哪幾個工具裡？
- 上線流程的「邊緣情況」誰處理？

**這份清單就是你 AI-Native 的 backlog**。每一條都要從某個人的腦袋搬到一個 agent 可以讀的地方。

### 2. 選一個 agent，但給它真的職責

別再做「萬能助理 GPT」了。選一個窄場景：

- 「處理新進客戶 onboarding 第一封 email 草稿」
- 「整理每週 sales pipeline 變化摘要」
- 「掃週末的 GitHub PR，找出可以 ship 的」

**一個 agent、一個 scope、三週測試期**。三週後 review：哪裡省時間？哪裡需要 fix？要不要 expand？

### 3. 把 Revenue per Employee 變成你的北極星

你的 deck 裡如果還沒有這個數字，加上去。**這是 Greg 文章說會「成為最清楚的訊號之一」的指標**。

你接下來每招一個人，問自己一個問題：

> 「這個工作能不能先用 agent 跑前 80%？我只招那 20% 真的需要人類的部分？」

這個問題能讓你的 burn rate 砍一半，**而且公司會反而更強**。

## 為什麼既有企業很難追上你（而你有窗口）

Greg 文章最後一段我特別愛：

> **「老公司不能靠宣布一個 AI 計畫就變 AI-Native。那就像想靠換方向盤把郵輪變快艇。」**

困難不在取得模型 — **每個人都能取得**。困難的是**既有企業背負大量老舊流程的債**：資料亂、政策衝突、團隊守地盤、工作流程以人力編制為中心設計、軟體堆疊用膠帶和季度規劃儀式拼起來。**整個作業系統預設人類是資訊的處理器**。

新公司（你）的優勢是**沒有家具要搬**。可以從乾淨的狀態開始。**每一個流程都用「agent 能不能先做第一輪？」這個問題來建**。從第一天就文件化。讓每一筆資料都可被使用。在錯誤變成災難之前就設計好人工審核點。

**這是早期新創在這個時代最大的 unfair advantage**。

## 結尾：那 1,000 家裡有幾家是台灣？

我不知道確切數字。但我猜亞洲不會超過 50 家，台灣不會超過 10 家。

**這就是機會**。

當所有人都在大聲講 AI，結構上準備好的公司極少。**這就是落差。這就是窗口**。

下一批偉大的公司，會是那些從內到外用 agent 重建資料、流程、政策和團隊的公司。它們會看起來比應有的規模小。它們的速度快到不合理。它們用更少的人做更有價值的工作。

> **它們會讓既有企業看起來像在跑一個有比較好看登入畫面的 Windows 95。**

大多數人還在問「怎麼在工作中用 AI」。你應該問的是：

> **「怎麼建一間 AI 能在裡面運作的公司？」**

這個問題就是那扇門。**現在幾乎沒有人走過去**。

## FAQ

**Q: 1,000 家這個數字怎麼算的？**

A: Greg Isenberg 自己也承認「具體數字不如結論重要」。他的判準是：年營收 ≥ 500 萬美金 + 核心工作流程為了讓 agent 執行而設計（不是邊緣加掛 AI）。500、1,000、2,000 都有可能，trend 是「**幾乎沒有人在做這件事**」。我自己的體感：年營收門檻拉掉，台灣真正算 AI-Native 的公司大概不超過 10 家。

**Q: 我們是 3 人新創，連基礎建設都沒有，要先做什麼？**

A: **你比上市櫃公司有優勢**。3 人新創的好處是沒有歷史包袱。從第一天開始：(1) 所有客戶資料丟一個地方（Notion / Airtable / Google Sheets 都可以），(2) 寫一份「公司決策政策手冊」（一頁就夠），(3) 第一個 hire 之前先問「這個能不能用 agent？」這三件事就會讓你領先 95% 的同齡新創。

**Q: AI-Native 跟 AI-First 是同一件事嗎？**

A: 不完全。AI-First 是策略宣示（「我們所有產品決策優先考慮 AI」）。AI-Native 是**結構性事實**（「我們公司的營運架構就是為了讓 agent 運作而設計」）。可以 AI-First 但不 AI-Native（很多大公司就是這樣，PPT 上是 AI-First，內部還是 Excel + email）。也可以反過來但比較少見。**Greg 用 AI-Native 是因為它指的是結構，不是宣言**。

**Q: Revenue per Employee 真的是定義性指標嗎？我們才剛 Pre-A，連 revenue 都不穩定。**

A: 早期可以用 proxy 指標：每位人類同事**負責的工作流數量**、每位同事**主導的 agent 數量**、**每月處理的客戶/案件數 / 人類團隊規模**。重點不是哪一個具體數字，是**讓「人類產出密度」變成一個你會追蹤的維度**。等你 Series A 之後，revenue per employee 會自然取代這些 proxy。

**Q: 我看完想做但團隊沒人做過 agent 系統，怎麼開始？**

A: 兩條路。(1) **自己土法煉鋼**：選一個窄場景，用 Claude API / OpenAI Agents SDK 建一個 MVP，三週見證。(2) **用現成平台**：例如 Omni / LangSmith / n8n 這類已經把 memory / skill / audit 都做好的平台，直接設定 agent 而不是寫 infra。對 3-5 人團隊，**我建議走第二條** — 別把工程資源花在重造 agent 基礎建設上。

**Q: Omni 跟 Greg 文章描述的 AI-Native 平台是什麼關係？**

A: **Omni 是把 Greg 文章描述的基礎建設產品化的平台**。Memory（per-conversation + Shared）對應 Greg 的「讓公司對機器可讀」；Skills 對應「把每個重複決策變成決策系統」；Subagents + Heartbeat + Monitor 對應「agent 艦隊 + 人類監督」；Plan Mode + Z App agent ticket 對應「知道何時動手、何時問人」；TAEA 對應 Greg 提到的「permissions and audit trails」。我們自己用 Omni 跑公司，**Omni 也是用 Omni 開發出來的**。
